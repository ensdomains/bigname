# Consumer Capability Baseline

Status: Phase 0 baseline

This document is the checked-in replacement contract for first-party consumers. The extracted apps monorepo reference under `.refs/apps_monorepo` informs app-facing coverage, but replacement is measured by bigname route capability coverage and cutover evidence rather than legacy schema parity.

## 1. Capability Groups

| Capability | Example consumer surface | Native `v1` responsibility |
| --- | --- | --- |
| exact name profile | profile pages, record editing, registration views | `Name.registration` + `Resolution` |
| names owned / controlled by address | dashboards and search flows | `Address.names` |
| names owned / controlled by address with role summary | dashboard lists | `Address.names` with `include=role_summary` |
| declared child subnames and counts | subname pages and creation flows | `Name.children` |
| record inventory for editing | profile and records screens | `Resolution.record_inventory` + `Resolution.record_cache` |
| verified record reads | profile, send, and address-resolution flows | `Resolution.verified_queries` |
| name history | profile history pages | `History(scope=both)` |
| address history across names | address activity views | `Address.history` |
| role holders for a resource | roles pages | `Permissions.by_resource` |
| role change history | roles history pages | `History(filter=permissions)` |
| resolver-centric overview | resolver pages | `Resolver` |
| claimed vs verified primary name | dashboard and profile | `PrimaryName.claimed_primary_name` + `PrimaryName.verified_primary_name` |
| app-facing compact name search and suggestions | Manager dashboard search and Explorer search | `GET /v1/names` |
| app-facing compact resolver records | Manager profile records and Explorer record panels | `GET /v1/resolve/{name}/records` or `GET /v1/names/{namespace}/{name}/records` |
| app-facing compact events | Manager / Explorer activity tables | `GET /v1/events` and history routes with `view=compact` |
| app-facing roles by account/resource/name | Explorer resolver and role pages | `GET /v1/roles`, `GET /v1/names/{namespace}/{name}/roles`, `GET /v1/resources/lookup` |
| app-facing compact resolver overview | Explorer resolver overview | `GET /v1/resolvers/{chain_id}/{resolver_address}/overview` |

## 2. Current Status

- this is the working baseline for Phase 0
- `Address.names` with `include=role_summary` is an additive expansion of the same address-to-surface collection; it is not a separate route or replacement surface
- `Address.history` is the declared-state address activity read over address-derived surface and resource anchors; it reuses the shared history contract rather than introducing a separate truth system
- `Resolution` is one mixed route: `record_inventory` defines the known record-selector space, `record_cache` is the declared last-known-value view over that same selector space, and `verified_queries` is the explicit request-bound execution answer set
- `GET /v1/resolve/{name}` and `GET /v1/resolve/{name}/records` are convenience entries to the same `Resolution` and compact records capabilities, not separate consumer capabilities: exact `base.eth` infers `namespace=ens`, names matching `*.base.eth` infer `namespace=basenames`, other supported ENS names infer `namespace=ens`, and responses still expose canonical namespaced identity where the envelope includes it. `GET /v1/resolve/{name}/records` defaults to `mode=auto`, using declared cache for authoritative resolver profiles and verified resolution for supported requested selectors when declared resolver-family coverage is pending; when no declared record selectors are available it probes only the bounded basic app profile set so common UI records can still appear. ENSv1 PublicResolver-compatible observed text selectors may be worker-hydrated into the declared cache after rebuild, so app reads can stay projection-local for those values. It reads current projections directly and does not run normalized-event catch-up scans for UI record rows.
- namespace-inferred resolution does not change verified-support meaning: inferred Basenames requests use Basenames-local selector and topology support, return selector-local `unsupported` outside the exact-surface transport-assisted direct-path support class, and do not fall back to ENS
- ENSv2 exact-name profile support is promoted only for the selected `sepolia-dev` deployment profile when `ens_v2_registrar_l1` declares `exact_name_profile = "supported"`; that support class covers declared exact-name profile reads from the admitted `ETHRegistry` and `ETHRegistrar` sources, and it does not graduate resolver-profile support, verified resolution, primary-name support, history coverage, other deployment profiles, or consumer replacement for unrelated capability groups (upstream: .refs/ens_v2/contracts/deployments/sepolia-dev/ETHRegistry.json:L2 @ ens_v2@554c309) (upstream: .refs/ens_v2/contracts/deployments/sepolia-dev/ETHRegistrar.json:L2 @ ens_v2@554c309) (upstream: .refs/ens_v2/contracts/src/registry/interfaces/IPermissionedRegistry.sol:L34 @ ens_v2@554c309) (upstream: .refs/ens_v2/contracts/src/registrar/interfaces/IETHRegistrar.sol:L32 @ ens_v2@554c309).
- ENSv1 and Basenames `Resolution.record_inventory`, `Resolution.record_cache`, and resolver-centric overview are not consumer-replacement complete merely because resolver discovery admits registry-observed resolver addresses. A current resolver target in registry state is insufficient by itself. For ENSv1, retained generic resolver-local record events may provide observed selector/cache facts while profile state is pending, but only after the indexed fields and ABI payload decode as the upstream resolver event shape; malformed topic collisions remain retained raw facts without selector/cache evidence. Complete family coverage, resolver overview completeness, latest-only behaviors, and event-to-onchain-call parity require supported ENS Labs PublicResolver-generation profile admission for the relevant family, and even supported shared PublicResolver targets do not enumerate current-name fan-in in resolver-overview `bindings`, `aliases`, or resolver event summaries; those sections remain explicit `UnsupportedSummary` with `unsupported_reason="resolver_binding_enumeration_not_projected"` while exact-name resolver state stays on exact-name routes (upstream: .refs/ens_v1/contracts/registry/ENS.sol:L12 @ ens_v1@91c966f) (upstream: .refs/ens_v1/contracts/resolvers/profiles/IAddrResolver.sol:L6 @ ens_v1@91c966f) (upstream: .refs/ens_v1/contracts/resolvers/profiles/IAddressResolver.sol:L6 @ ens_v1@91c966f) (upstream: .refs/ens_v1/contracts/resolvers/profiles/ITextResolver.sol:L5 @ ens_v1@91c966f) (upstream: .refs/ens_v1/contracts/resolvers/profiles/ITextResolver.sol:L10 @ ens_v1@91c966f). ## 4. Exit Condition For Consumer Cutover

Before first-party cutover:

- each capability must map to one or more concrete app call sites
- each capability must have contract tests
- each capability must have rollout and rollback criteria
