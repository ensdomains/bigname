## 1. apps/api

### Overall assessment

The apps/api subsystem is in strong shape relative to its docs: the route catalog matches `docs/api-v1.md` exactly (all 24 documented routes plus `/`, `/docs`, `/openapi.json` are registered in `apps/api/src/openapi/server.rs:45-55`), the hard snapshot invariants are enforced (`at` + `chain_positions` → `invalid_input` in `crates/storage/src/snapshot_selection/selection.rs:35-39,71-75`; stale → 409 `stale` via `support/snapshots.rs:330-343`), unsupported sections are consistently rendered as explicit `UnsupportedSummary` objects rather than omitted, ResultStatus field boundaries on the primary-name route are aggressively validated (`support/primary_name_readback.rs:271-332`), and cursor envelopes are versioned, route/filter-bound, and replay-validated. The main weaknesses are: the primary-names route-level coverage classifier diverging from documented class semantics, a documented-but-unimplemented `/v1/names` expansion, `meta=full` being a no-op on several compact routes, one hardcoded coverage object, and a cluster of "load everything, paginate in memory" patterns on history/events routes.

### GAPS

- **[moderate/high] apps/api/src/responses/app_facing/names_collection.rs:21-40 — `include=record_summaries` and the `record_summaries`/`subname_count`/`record_count` fields of `CompactDomainSummary` are not implemented.** Docs (`api-v1-routes.md:349`, `api-v1.md:321`) describe `include=record_summaries` as adding record counts, known text-key hints, avatar/content-hash presence, and coin-type hints. The handler accepts the flag but unconditionally routes it into `meta.unsupported_fields` (`handlers/app_facing/names_collection_support.rs:89-93`), and the item builder never emits any of the three fields. `record_summaries` is at least disclosed; `subname_count`/`record_count` are silently absent, which conflicts with the versioning rule "an unsupported capability is reported through coverage or error; never silent omission" (`api-v1.md:446`). Fix: either implement from declared inventory/cache (the data already exists for the children/address routes) or add the missing fields to `unsupported_fields` unconditionally and amend the docs.
- **[moderate/high] apps/api/src/handlers/app_facing/names_collection.rs:40-47, apps/api/src/responses/app_facing/roles.rs:39-52,75-86 — `meta=full` is a no-op on `/v1/names`, `/v1/roles`, `/v1/names/{ns}/{name}/roles`, and `/v1/resources/lookup`.** `api-v1.md:147` says `meta=full` "adds the full-envelope coverage, chain_positions, consistency, last_updated, and route-level provenance summaries." Records, events, children, and resolver-overview implement this (e.g. `responses/app_facing/records_value_meta.rs:113-136`, `responses/app_facing/events.rs:93-108`), but the names and roles family render the identical summary object for `summary` and `full`. Fix: add the full-envelope block in those builders or scope the doc claim per route.
- **[minor/high] apps/api/src/handlers/app_facing/names_collection_support.rs:65-75 — `resolved_address` is unconditionally rejected with `400 unsupported`.** Docs (`api-v1-routes.md:347`) frame it as conditionally supported "where a declared, replay-stable record-value equality projection exists." Since no projection exists anywhere, blanket rejection is technically in-contract, but the parameter is advertised in OpenAPI (`routes/parameters.rs:304-308`) with no supported case at all — worth a doc/openapi note that it is currently always unsupported.

### INCORRECT BEHAVIOR

- **[significant/medium-high] apps/api/src/responses/resolution.rs:467-507 — primary-names route coverage misclassifies persisted exact-tuple reads.** `primary_name_route_coverage` only returns the documented "ENS/Basenames mainnet supported tuple class" coverage (`status=partial`, `["ens_v1_reverse_l1","ens_execution"]`) when the tuple row is present **and** `persisted_verified.is_some()`. Since `persisted_verified` is only loaded when `mode` includes verified (`support/primary_name_lookup.rs:19-24`), a default `mode=declared` read that successfully serves a persisted `primary_names_current` claim returns the *out-of-class* coverage object (`status=unsupported`, "exact-tuple persisted readback is not supported for the requested tuple"). Same for `mode=both` when no verified outcome row exists yet. `api-v1-routes.md:630-635` says route coverage "describes whether the answer came from the persisted exact-tuple class", and a persisted declared claim plainly did. The route tests never assert coverage for this case (`tests/primary_names.rs:705-792` asserts only declared/verified state), so this is not pinned intentionally. Fix: classify on tuple-class membership (namespace/coin-type + projection availability), not on the presence of a persisted verified outcome.
- **[moderate/medium] apps/api/src/responses/resolution.rs:369-381 — `verified_primary_name` returns `unsupported` for an in-class tuple with no persisted verified outcome.** For `TuplePresent` without persisted readback the route emits `status=unsupported, "verified primary-name entrypoint is not yet supported"`. Docs say "No declared or verified answer for the tuple returns `200` with `status=not_found`" and reserve `unsupported` for out-of-class entrypoints (`api-v1-routes.md:623-624,633`); the ENS mainnet tuple class includes `ens_execution` as a supported verified source. Tests pin the current behavior (`tests/primary_names.rs:780-788`), so this is a deliberate code/doc divergence — one of the two needs to move.
- **[moderate/high] apps/api/src/responses/collections.rs:52-58 — `/v1/addresses/{address}/names` coverage is hardcoded.** Every response gets `status=full, exhaustiveness=authoritative, source_classes_considered=["ensv1_registry_path"]`, including `namespace=basenames` reads whose declared truth comes from the Base authority split (`api-v1-routes.md:423`), and including non-ENS rows in unscoped reads. The storage summary carries no coverage (`crates/storage/src/address_names/types.rs:120-126`), so the API invents an inaccurate one. Fix: derive source classes from row namespaces/manifests or at minimum split by namespace filter.
- **[moderate/medium] apps/api/src/handlers/exact_name.rs:3-27 — exact-name path segments are never normalization-validated.** `api-v1.md:12` says path-segment names are ENSIP-15-normalized and "unnormalizable names reject as `invalid_input`". The canonical name/coverage/explain/children/records/roles/history routes build `logical_name_id = "{ns}:{name}"` from the raw decoded path, so an unnormalizable (or merely unnormalized, e.g. `Alice.eth`) input falls out as `404 not_found` rather than `400 invalid_input`. The profile/identity routes normalize correctly (`support/records.rs:49-69`). Fix: run the same normalization boundary on path names and reject unnormalizable input with 400 (lookups for normalizable-but-unnormalized input can stay 404 or be canonicalized, but pick one and document it).
- **[minor/high] apps/api/src/support/cursors.rs:330-336 — stale/invalid cursors return code `invalid_input`, docs say `400 invalid_cursor`.** `api-v1.md:418` promises `invalid_cursor` for cursors invalidated by label-preimage repairs, but `invalid_cursor` is also absent from the documented error-code list (`api-v1.md:432`), so the docs are internally inconsistent; code should win one way or the other.
- **[minor/high] apps/api/src/responses/projections/provenance.rs:18-22 — declared-only provenance emits `execution_trace_id: null`.** Docs (`api-v1.md:198`) say declared-only provenance *omits* `execution_trace_id`. Same pattern in `responses/collections.rs:308`.
- **[minor/high] apps/api/src/support/query_parsing.rs:106-124 — primary-name `coin_type` is not canonicalized.** `"060"` or `"0060"` passes digit validation, forms distinct tuple keys (`primary_name_verified_request_key`, `support/primary_name_lookup.rs:343-349`), silently misses persisted `coin_type="60"` rows, *and* disables the ENS/60 fallback (`coin_type != "60"` check at `support/primary_name_lookup.rs:357`), yielding out-of-class `not_found` for what the caller meant as coin type 60. Strip leading zeros (or parse to integer and re-render) before lookup.
- **[minor/medium] apps/api/src/responses/app_facing/records.rs:180-234 + 297-338 — unrequested compact-record sections render as `null`, colliding with documented null semantics.** `data.resolver_address` is `null` both when the section wasn't requested (e.g. `include=avatar` resets the default include set) and when "no declared resolver at the snapshot" (`api-v1-routes.md:385`). A client can't distinguish the two. Omit unrequested sections instead.
- **[minor/medium] apps/api/src/responses/app_facing/identity.rs:339-361 — `/v1/status` can return `"stale"`, which is not a documented status, and the early `return "stale"` short-circuits the degraded check** (a later chain with missing readiness data should produce `degraded` per `api-v1-routes.md:327`). Order-dependent and undocumented vocabulary.
- **[minor/high] Inconsistent address validation across routes.** `/v1/names?owner=`, `/v1/primary-names/{address}` reject malformed addresses with 400 (`names_collection_support.rs:353-369`, `query_parsing.rs:81-91`), while `/v1/addresses/{address}/names` (`handlers/collections.rs:13`), `/v1/history/addresses/{address}` (`handlers/history.rs:14`), and `/v1/events?address=` (`handlers/app_facing/events.rs:69`) lowercase any garbage via the lenient `normalize_address` fallback (`query_parsing.rs:409-411`) and return empty 200s. Same input, three behaviors.
- **[minor/medium] apps/api/src/support/query_parsing.rs:17-28 — `mode=` (empty string) returns 400 on primary-names/profile while `view=`/`meta=` empty values fall back to defaults** (`query_parsing.rs:30-47,64-79`). Inconsistent empty-value handling; same for `name` not being trimmed on `/v1/names` while `prefix`/`contains` are (`names_collection_support.rs:313-351`).

### OPTIMIZATIONS

- **[significant/high] Unbounded full-history loads with in-memory pagination on `/v1/events`, `/v1/history/names|resources|addresses`.** `load_event_history`/`load_name_history`/`load_address_history` have no LIMIT (`crates/storage/src/history.rs:95,139,244`); handlers fetch the entire filtered set per request, then slice (`handlers/history.rs:16-38,107,185-197`; `handlers/app_facing/events.rs:20-41`; window scan in `support/cursors.rs:33-70`). An unfiltered `/v1/events?page_size=50` reads every canonical event each call; `paginate_window` also does an O(n) scan to find the cursor. Push keyset pagination into SQL like the names/children/permissions routes already do.
- **[significant/high] /v1/names runs a full `COUNT(*)` over the filtered CTE on every request** regardless of `include=total_count` (`crates/storage/src/name_current/list.rs:187,253-270`; only exposed when requested at `handlers/app_facing/names_collection.rs:43`). For the documented suggestion/search latency path (`prefix=...&meta=none`) this doubles the query cost; gate the count on the include flag.
- **[moderate/high] apps/api/src/handlers/collections.rs:244-278 — N+1 `load_name_surface` calls in the compact children view.** With `include=counts`, one sequential query per child row on every page (on top of the already-batched `load_name_current_by_logical_name_ids` and summaries); batch it like the other lookups in the same function.
- **[minor/high] apps/api/src/responses/json.rs:1-10 + responses/collections.rs:321-334 — O(n²) `Vec::contains` dedupe** in history/collection provenance builders, executed over the full (unpaginated) row set for `view=full`/`meta=full`; use a set.
- **[minor/medium] crates/execution/src/ens_primary_name.rs:166-179 — a fresh `JsonRpcHttpClient` (reqwest client) is constructed per primary-name fallback request** (invoked per API call from `support/primary_name_lookup.rs:361`); cache it in `AppState`. Also the tuple lookup and persisted verified readback in `load_primary_name_lookup_state` are sequential awaits that could be joined (`support/primary_name_lookup.rs:10-25`).

### Three biggest themes

1. **The primary-names route's coverage/status classifier is the largest doc-vs-code divergence**: route-level coverage keys off "persisted verified outcome present" instead of documented tuple-class membership, and verified misses for in-class tuples surface as `unsupported` rather than `not_found` — both make the route's contract signals unreliable for clients deciding whether to trust an answer.
2. **Compact-route polish is uneven**: documented expansions (`record_summaries`, `meta=full` envelope summaries) are accepted-but-inert on the names/roles family while fully implemented on records/events/children/resolver-overview, and input normalization (addresses, empty enum values, trimming, path-name normalization) differs route by route — the shared parsers exist but aren't applied uniformly.
3. **Pagination is split between two generations**: names/children/permissions/roles use proper SQL keyset pagination with cursor existence checks, while history/events still load the entire filtered universe per request and paginate in memory (plus an always-on COUNT on `/v1/names`) — the clearest scalability cliff in the subsystem.

## 2. apps/indexer

### Overall assessment

The intake plane is carefully built and matches most documented promises: hash-first fetch with block-scope identity checks everywhere (`provider/decode.rs:94-132`, `block_transaction.rs:28-72`), explicit reorg unwind with orphan marking and execution-cache invalidation (`reconciliation/canonical/orphaning.rs`), monotonic safe/finalized promotion gated on canonical-branch ancestry proofs (`canonical/checkpoints.rs:81-129`, `crates/storage/src/checkpoints.rs:246-285`), per-profile provider availability with explicit `unavailable/no_provider` idling (`runtime/intake.rs:141-184`, `bootstrap_backfill.rs:141-152`), selected-target-only backfill with effective-range interval filtering (`backfill/selection.rs`), capacity preflight with persisted failure metadata (`ops_catchup/capacity.rs`), and backfill range checkpoints that never touch chain checkpoints (no backfill path calls `advance_chain_checkpoints`; resume is correctly `checkpoint+1`, `reservation_execution.rs:478-481`). The Coinbase SQL path implements both validation modes, hash-pins SQL log identities against the validation provider, and forces raw-only mode for Basenames authority families as documented. The significant problems are concentrated in three places: backfill lease lifecycle under concurrency, canonicality assignment using stale stored anchors during a live reorg, and a fail-fast/no-retry posture that makes long-running intake fragile.

### GAPS

- **[significant/medium] apps/indexer/src/main/backfill/fetching/canonicality.rs:117-150 + reservation_execution.rs:484-505 — reorg-during-backfill can re-canonicalize orphaned blocks.** `BackfillCanonicalityEvidence` is loaded once per reserved range and its stored canonical anchor is then used for the whole range. `apply_stored_anchor` proves "canonical" via `chain_lineage_contains_ancestor`, which walks parent links *ignoring canonicality state* (`crates/storage/src/lineage/reads.rs:182-235`), and the lineage upsert explicitly revives `orphaned → canonical` (`crates/storage/src/lineage/upserts.rs:127-128`). Sequence: chunk resolves number→hash H; live poll reorgs and orphans H; backfill's fresh provider revalidation skips H (hash mismatch) but the stale stored anchor (the now-orphaned old head, within the 1024-block window) still proves ancestry → H's lineage/raw facts are upserted back to `canonical` with no later repair pass, since live reorg repair already completed. Docs require reorg repair to be the only path that flips canonicality. Fix: reload checkpoint evidence per chunk and/or have `chain_lineage_contains_ancestor` (or `apply_stored_anchor`) reject paths through `orphaned` rows.
- **[moderate/medium] apps/indexer/src/main/backfill/fetching/sparse.rs:213,291-295 — raw-only sparse backfill never fetches code observations.** `let code_observation_block = None;` is hardcoded, so the `SparseCodeObservationPlan` machinery is dead and `raw_code_hash_count` is always 0 on this path. This is the default path for automatic bootstrap (`run_mode.rs:38-39` maps Auto→RawOnly at startup). Bootstrap-admitted history therefore has no `raw_code_hashes`, while live intake (`reconciliation/persistence.rs:296-362`) and the Coinbase path (`fetching/historical.rs:158-163`) do persist them. Docs list code-hash observations as in-scope intake facts, and ENSv1 resolver-profile admission may rely on stored code-hash observations.
- **[moderate/high] apps/indexer/src/provider/logs_receipts.rs:480-483 — log-range splitting only recognizes one provider's limit error.** `is_log_range_result_limit_error` matches only `"query exceeds max results"`. Geth ("query returned more than 10000 results"), Infura, Alchemy ("Log response size exceeded"), and Erigon emit different messages, so on those providers an oversized `eth_getLogs` range hard-fails the backfill range instead of bisecting (`logs_receipts.rs:280-296`).
- **[moderate/medium] apps/indexer/src/provider/block_transaction.rs:100-136 — providers that error (rather than return null) on `safe`/`finalized` tags break head-following entirely.** The head snapshot models absent safe/finalized as `Option::None`, but only a JSON `null` result maps to `None`; a JSON-RPC error for the `safe` tag fails the whole batch, then the per-call fallback fails again (`request.rs:98-115`), so `fetch_chain_heads` errors and canonical head-following stops too. Docs say such sources should degrade to bootstrap/shadow use, not break intake.
- **[moderate/medium] apps/indexer/src/main/reconciliation/canonical.rs:537-609 + persistence.rs:75-156 — live gap-fill is unbounded and non-checkpointed.** The contiguous-gap fast path accepts gaps up to `MAX_PARENT_FETCH_DEPTH = 131_072` (canonical.rs:38,550) and then `persist_reconciled_raw_payloads` fetches a full bundle per gap block *sequentially* (`block_transaction.rs:283-298`), accumulating all transactions/receipts/logs in memory before upserting, with the chain checkpoint advancing only at the very end. After extended downtime this is hours of un-checkpointed single-poll work; a crash redoes all of it. Docs scope the recent window to "short parent gaps" with bounded jobs owning history; there is no cutoff that hands a large gap to the bounded backfill machinery.
- **[minor/high] apps/indexer/src/main/reconciliation/canonical.rs:418-491 — no escalation path for reorgs deeper than the walk limit.** A reorg/ancestor gap that exceeds `MAX_PARENT_FETCH_DEPTH` yields `AwaitingAncestor` forever; every poll repeats the full walk (one `load_chain_lineage_block`/`fetch_block_by_hash` per ancestor) with no operator-facing fail-closed signal beyond a status log; recovery requires knowing to run `rewind`.
- **[minor/high] apps/indexer/src/main/ops_catchup/capacity.rs:130-159 — object-cache budget is never checked.** Metadata hardcodes `"object_cache_budget_checked": false`; docs say the preflight covers "any configured object-cache budget". Explicit, but still an unimplemented documented check.

### INCORRECT BEHAVIOR

- **[significant/medium-high] backfill leases expire mid-range and are then stolen, aborting the whole job.** Leases are set only at reservation (`reservation_execution.rs:410-417`) and never refreshed during the chunk loop — `advance_backfill_range` updates checkpoint/status but not `lease_expires_at` (`crates/storage/src/backfill_jobs/lease.rs:197-213`). Bootstrap uses a 300 s lease (`bootstrap_backfill.rs:48`) on 50,000-block child ranges (`bootstrap_backfill.rs:50`), which cannot finish in 300 s. With the default 4 concurrent workers (`bootstrap_backfill.rs:51,499-508`), `reserve_backfill_range` *prefers* expired reserved/running ranges over pending ones (`lease.rs:58-86`), so worker B steals worker A's active range; A's next `advance` fails the lease check (`validate.rs:152-161`), its `fail_backfill_range` also fails the lease check, and the worker error triggers `abort_all` (`concurrent_execution.rs:109-119`) — killing the entire bootstrap and the `run` command (`run.rs:64-76`). Result: duplicate fetching plus a crash-looping startup on any deployment where a range takes >300 s. Fix: refresh `lease_expires_at` on every `advance_backfill_range`, or size lease duration to range duration.
- **[moderate/high] apps/indexer/src/main/reconciliation/persistence.rs:71,158-176 + canonical.rs:216-313 — the documented one-block atomic admission unit is not one transaction.** Docs (chain-intake.md "Atomicity boundary") require one transaction containing the lineage anchor, raw facts, normalized events, invalidation signals, with the head pointer written last inside that unit. The implementation issues separate pool-level upserts (`upsert_raw_blocks`, then cache metadata, transactions, receipts, logs, event-silent rows, adapter sync) each in its own implicit transaction, with `advance_chain_checkpoints` as a later separate transaction. Ordering is right and recovery works (checkpoint not advanced ⇒ next poll re-persists), but partially admitted blocks (e.g. transactions persisted, logs not) are visible to downstream readers mid-write and after a crash until the next poll.
- **[moderate/medium] live vs backfill raw-log retention diverges for the same block.** Live admission retains *all* logs of any transaction containing a selected log (`reconciliation/payload.rs:135-155`, used at persistence.rs:121-126); hash-pinned/sparse/Coinbase backfill retains only selected-emitter logs (`backfill/fetching.rs:237-242`, `sparse.rs:283-289`, `historical.rs:123-128`). Docs promise backfill "uses the same stages/downstream semantics as live ingestion"; a block admitted via backfill permanently lacks tx-sibling raw logs that the same block admitted live would retain, so later replay sees different selected-fact sets depending on which path admitted the block.
- **[moderate/high] apps/indexer/src/main/runtime/poll_loop.rs:356-361,369-374 — transient DB errors kill the whole poll loop.** `normalized_replay_cursors_complete(...).await?` and `sync_live_adapter_backlog_after_normalized_replay(...).await?` propagate out of `run_poll_loop`, exiting the indexer process, while every other refresh failure in the same loop is caught and logged ("keeping last successful state"). One transient Postgres hiccup at the wrong tick terminates live intake.
- **[minor/medium] apps/indexer/src/main/ops_catchup/planning.rs:46-108 + runner.rs:173-178 — moving tail chunk causes overlapping re-execution each follow iteration.** The final chunk's end is the current finalized head and the idempotency key embeds `to=`, so each follow iteration creates a *new* job whose range overlaps the prior iteration's completed tail chunk, re-fetching those blocks (idempotent upserts, so no corruption — wasted work and job-row growth). Fixed-stride chunk starts keep earlier chunks reusable, but the tail is redone every iteration.

### IMPROVEMENTS

- **[moderate/high] apps/indexer/src/provider/request.rs:177-221 — no retry/backoff in the primary JSON-RPC client.** Single attempt per request; the only "retry" is batch→sequential-single fallback with zero delay (request.rs:93-115), which makes a 429/rate-limit failure *worse* by immediately firing N individual calls. Combined with fail-fast range handling (`failure_recording.rs`), any transient provider error during a multi-hour backfill marks the range/job failed and (at startup) exits the process. Contrast: the Coinbase SQL client has 5 attempts with backoff (`backfill/coinbase_sql/client.rs:19,52-92`) and the tx/receipt path has a 3×250 ms retry (`provider/transaction_receipts.rs:24-25`). The chain provider deserves at least equivalent treatment.
- **[moderate/high] apps/indexer/src/main/run.rs:64-76 — startup bootstrap is serial with, and fail-fast for, live intake.** Full-admitted-history bootstrap (days on a fresh mainnet deployment) must drain before `run_poll_loop` starts, so head-following and checkpoint rows stay idle the whole time; any single range failure aborts `bigname-indexer run`. Consider running bootstrap concurrently with head-following (the job substrate already supports it) and retrying failed ranges in-process.
- **[moderate/high] triplicated materialization pipelines.** `backfill/fetching.rs:57-373` (inline), `fetching/sparse.rs:198-455` (raw-only), and `fetching/historical.rs:43-237` (Coinbase) each reimplement the block loop, selected-address filtering, identity checks, and upsert ordering. They have already drifted (code observations: per-selected-address vs none vs log-emitters; logs upserts with vs without snapshots). The dead `ensure_provider_bundle_matches_raw_block` self-check at fetching.rs:215 (comparing a bundle to a raw block derived from that same bundle) is a symptom.
- **[minor/high] apps/indexer/src/main/backfill/coinbase_sql/client.rs:81-84 — intermediate retry errors are discarded.** The `Err(error)` arm for retryable attempts ignores `error`; only the final attempt's error surfaces, losing the earlier failure context (e.g. whether retries were 429 vs network).
- **[minor/medium] apps/indexer/src/main/ops_catchup/config.rs:10 — `DEFAULT_OPS_CATCHUP_CHUNK_BLOCKS = 32`.** Full-history catch-up from ENSv1's start (~3.3 M → ~22 M+) at 32-block chunks means ~600 k jobs, each with a source-plan load, job upsert, and capacity check per follow iteration even when already completed (`runner.rs:157-194`). The default is pathological for the documented "full admitted history" use.

### OPTIMIZATIONS

- **[moderate/high] apps/indexer/src/main/reconciliation/persistence.rs:296-362 + payload.rs:57-77 — code observations for every watched address at every reconciled block.** Every changed head block (and *every* block of a live gap-fill, since all reconciled blocks count as "refreshed") triggers `eth_getCode` for the full chain watch plan (`task.addresses` — with discovery-expanded ENSv1 resolvers this is large). A 7,200-block gap × N watched addresses is millions of calls. Restrict to log-emitters (as the Coinbase path does) or to blocks where the address set changed.
- **[moderate/high] apps/indexer/src/main/backfill/fetching.rs:162-176 — inline mode (used by ops-catchup, runner.rs:187) fetches full transaction+receipt bundles for every block in the chunk** (`fetch_block_bundles_without_logs_by_hashes` = full block body + `eth_getBlockReceipts` per block) even though ranged logs already identify the blocks that matter; the raw-only sparse path proves the logs-first + targeted `eth_getTransactionByHash/Receipt` shape works (`sparse.rs:304-336`). Over full history this is the dominant RPC cost.
- **[minor/high] apps/indexer/src/main/reconciliation/persistence.rs:179-242 — per-row INSERT loop for event-silent resolver observations** (one round-trip per row) where every other fact family uses bulk upserts.
- **[minor/high] apps/indexer/src/main/reconciliation/replay.rs:212-337 — three sequential scans of the same `raw_logs` join** (count, block hashes, emitter addresses) per replay request; one pass could produce all three.
- **[minor/medium] apps/indexer/src/provider/block_transaction.rs:478-496 — head blocks fetched by hash one-at-a-time** after the tag batch (up to 3 extra sequential round-trips per poll); and a safe/finalized head change re-fetches the full bundle of a block whose raw facts are almost always already persisted (`payload.rs:40-52` → `persistence.rs:110-114`), with no already-persisted skip.
- **[minor/medium] apps/indexer/src/provider.rs:41-43 — batch concurrency defaults to 1**, so chunked batch requests (headers, hash resolution) within a 1,024-block chunk run strictly sequentially; bumping `BIGNAME_INDEXER_JSON_RPC_BATCH_CONCURRENCY` is left entirely to the operator with no adaptive behavior.

### Three biggest themes

1. **Lease/lifecycle and staleness races around long-running backfill** — the never-refreshed 300 s lease vs 50k-block ranges (worker self-disruption, job aborts) and the per-range stale canonicality anchor (orphaned blocks revived to canonical during a live reorg) are the two findings most likely to cause real incorrect state or operational crash-loops, and both stem from loading coordination state once and holding it too long.
2. **Fail-fast with no transient-error tolerance** — the JSON-RPC client has zero retry/backoff, backfill ranges and the startup `run` command fail terminally on the first error, and the poll loop exits on transient DB errors in the replay-gating path; for a system designed around multi-day historical drains, transient faults are the common case and the code treats them as fatal.
3. **Three diverged copies of the materialization stage** — inline, raw-only sparse, and historical/Coinbase pipelines have already drifted on code observations (one path silently disabled), raw-log retention (live vs backfill sibling-log mismatch), and fetch strategy (full bundles vs logs-first); consolidating them behind one stage implementation would close several gaps in this report at once.

## 3. apps/worker

### Overall assessment

The worker subsystem is well-structured and largely faithful to the docs: the projection-apply queue implements generation-guarded, lease-based claims with dependency-ordered apply; bootstrap replay/handoff matches the documented cursor-seeding protocol (watermark captured before replay, seeded after, `automatic_projection_replay.rs:336-368`); both hydration passes are hash-pinned to stored checkpoints and never use provider `latest`; inspect tools use a read-only connection (`inspect.rs:116-131`); manifest-drift audit writes only its alert table. Every projected row I checked carries provenance, manifest version, chain positions, and canonicality summaries. The significant problems are concentrated in three places: documented invalidation/consumption paths that don't exist (fuse/scope modifiers, manifest-driven invalidation, ENSv2 `ParentChanged`), missing poison/dead-letter handling, and a multi-worker stale-overwrite window the generation guard does not actually close.

### GAPS

1. **[significant/high] apps/worker/src/permissions/load.rs:23,72 + apps/worker/src/permissions/project.rs:49** — `permissions_current` rebuild consumes only `PermissionChanged`; `PermissionScopeChanged` is never read, and no fuse-masking code exists anywhere in the worker (grep for fuse/scope handling returns only `derive_queries.rs:160,308`). projections.md:115 says "Workers apply the latest canonical scope modifier at or before the row's chain-position context after selecting the current `PermissionChanged` row," and :119 requires fuse application to mask `effective_powers` ("A burned fuse never appears as an available public power"). The adapter emits only `PermissionScopeChanged` on fuse changes (`crates/adapters/src/ens_v1_unwrapped_authority/apply_wrapper.rs:111,174`), so the derive-enqueued invalidation (`derive_queries.rs:160`) triggers a rebuild that reproduces the identical unmasked row — the invalidation is a structural no-op and overbroad powers stay published. Fix: load `PermissionScopeChanged` per resource in `load_permission_events`, apply the latest modifier when projecting each row, and include it in provenance/chain positions. (`RootPermissionChanged` from the taxonomy is likewise consumed nowhere, but the adapters don't emit it either — an upstream gap worth tracking.)
2. **[significant/high] no manifest-driven invalidation producer exists** — projections.md:195-202 lists "manifest version change that affects a consumed capability" as an invalidation trigger and says non-normalized-event producers "enqueue directly into `projection_invalidations`." The only writers in the repo are the worker's normalized-event derive, storage repair paths, label-preimage import, and the surface-binding trigger (grep over `INSERT INTO projection_invalidations`). Projections read manifest state at rebuild time — e.g. Basenames transport admission `record_inventory/chain_position.rs:81-113`, `name_current.rs:214` — so a manifest rollout/capability flip leaves rows stale until an unrelated event happens to invalidate the key. Fix: a manifest-change producer (trigger or worker loop) that enqueues affected family keys.
3. **[significant/high] no poison/dead-letter handling** — `attempt_count` is incremented at claim (`projection_apply/apply.rs:323`) but never read; `fail_invalidation` (`apply.rs:483-517`) just resets the claim and the row retries forever every `FAILURE_RETRY_DELAY` (60s, `projection_apply.rs:19`). A deterministically failing key (e.g. the 120s `name_current` timeout at `apply.rs:18,200-216` firing on a pathological name) loops indefinitely, and a poisoned `primary_names_current` key permanently blocks legacy reverse hydration (`projection_apply.rs:121-124`, per projections.md:173). Fix: max-attempt threshold routing into a dead-letter state with operator visibility.
4. **[moderate/high] ENSv2 `ParentChanged` never invalidates `children_current`** — the adapter emits it with `logical_name_id = None` (`crates/adapters/src/ens_v2_registry/events.rs:465-487`), so derive branch 1 produces a NULL key (filtered by `UPSERT_SUFFIX` at `derive_queries.rs:24`) and branch 2's join on `after_state->>'parent_node'` (`derive_queries.rs:112`) only matches ENSv1-shaped payloads. Yet the rebuild consumes `ParentChanged` as a primary input (`crates/storage/src/children/sources.rs:255-292`, `$10`). A re-rooting `ParentChanged` leaves stale child rows until a full replay. projections.md:99,202. Fix: derive a children key from `registry_contract_instance_id`/`parent_contract_instance_id` payload fields.
5. **[moderate/high] range rebuild and rewind are not implemented** — projections.md:210 promises "point rebuild by key, range rebuild by chain position, and full rebuild" for every projection, and :214-218 makes rewind-to-`ChainPositions` (with snapshot-scoped materialization for older snapshots) worker-owned. All rebuild CLI args are key-only (`cli.rs:230-522`); there is no chain-position/range/`at` argument and no rewind or snapshot-materialization code in the worker (grep). Only point and full rebuild exist.
6. **[moderate/high] per-family one-shot rebuild commands don't clear replay markers** — projections.md:230: one-shot rebuilds "clear any stale marker before rebuilding so a failed rebuild cannot leave a misleading completion marker behind." Marker clearing happens only inside the replay orchestrator (`replay/rebuild.rs:202-222`); `commands.rs:116-235` family commands never touch `current_projection_replay_status`. Combined with clear-then-rebuild full rebuilds (finding 12), a crash during `bigname-worker permissions-current rebuild` leaves a partially empty table while bootstrap skip logic (`automatic_projection_replay.rs:384-406`) still sees a complete marker set and skips replay — partial state persists indefinitely.
7. **[minor/medium] inspect surface is narrower than documented** — architecture.md operations promises inspect over "execution traces, manifest drift / proxy alerts, surface bindings, resolver topology, raw facts, manifest versions" plus "diff declared vs verified." Shipped subcommands (`inspect.rs:28-43`): backfill-job, canonicality, execution-trace, manifest-drift, stored-lineage-range, watch-plan. Missing: surface-binding inspect, resolver-topology inspect, raw-fact browsing beyond per-block counts, manifest-version listing, and any declared-vs-verified diff tool. The `inspect canonicality` contract itself is fully met (`inspect/canonicality.rs:30-53`).
8. **[minor/low] alias/linked/wildcard children buckets absent** — `children.rs` only builds `surface_class='declared'` (`children.rs:16,190`) and derive has no `AliasChanged`/`WildcardCoverageChanged` children path (`derive_queries.rs:80-140`); projections.md:28 lists alias/wildcard as children source events and :101 describes the buckets as distinct classes (only counts deferred). Internally consistent (no rows → no invalidation needed), so likely deliberate deferral, but it diverges from the family table.

### INCORRECT BEHAVIOR

9. **[significant/medium] stale-overwrite race between workers; generation guard protects the queue, not the table** — `UPSERT_SUFFIX` clears `claim_token`/`claimed_at` when a new generation lands on an in-flight key (`derive_queries.rs:52-55`), making it instantly claimable. Worker B can claim gen G+1, rebuild with fresh events, and delete the queue row while worker A's slower gen-G rebuild is still running; A's later upsert (plain upsert, no per-key lock — e.g. `name_current.rs:188-208`) then publishes the stale row last, with nothing left in the queue to repair it. projections.md:202's "a newer change cannot be lost by an older rebuild finishing late" holds for the queue row only. Multi-worker deployments only. Fix: keep the claim on generation bump (let the holder finish, then the surviving row re-applies), or take a per-key advisory lock around rebuild+complete.
10. **[moderate/high] only the actively-applied row is heartbeated** — a claim takes up to 25 rows (`projection_apply.rs:18`, `apply.rs:47`), but `spawn_claim_heartbeat` covers only the row currently in `apply_one_with_claim_heartbeat` (`apply.rs:162-175`). With slow rebuilds (`name_current` up to 120s each), trailing claimed rows exceed `CLAIM_RETRY_DELAY` (5min, `projection_apply.rs:20`) before being reached and become reclaimable by another worker — duplicate rebuilds and a wider window for finding 9. Fix: heartbeat the whole claimed batch, or shrink the batch.
11. **[moderate/medium] cross-family starvation in the claim order** — `claim_pending_invalidations` orders by family rank before `last_changed_at` (`apply.rs:298-316`), so a sustained `name_current` backlog larger than the 25-row batch indefinitely defers `address_names_current`/`primary_names_current` work; primary hydration is additionally blocked the whole time (`projection_apply.rs:88-138`). Dependency order per batch is documented, but unbounded cross-batch priority is not. Fix: age-based escalation or per-family quota in the claim.
12. **[moderate/high] clear-then-rebuild full rebuilds expose empty/partial projections** — `permissions.rs:51`, `children.rs:40`, `record_inventory/projection.rs:52`, `resolver.rs:97`, `primary_name/projection.rs:41` all delete the entire table up front and repopulate incrementally over a long scan, while `name_current` (`name_current.rs:120-177`, `NameCurrentReplacement`) and `address_names_current` (`address_names/rebuild.rs:86-146`, staged tables with atomic publish) do it correctly. A CLI full rebuild on a live deployment wipes reads for the duration, and a crash leaves partial state (see finding 6). Fix: extend the staged-replacement pattern to the remaining five families.
13. **[minor/medium] inconsistent NULL `log_index` tie-breaking across families** — permissions sorts `log_index ASC NULLS FIRST` (`permissions/load.rs:76-78`), address_names coalesces to `2147483647` i.e. last (`address_names/load.rs:263-266`), record_inventory's in-memory key uses `i64::MIN` i.e. first (`record_inventory/loading.rs:101-107`), derive uses `-1` (`derive_queries.rs:197,253`). Each family is internally deterministic, but two families can disagree about which same-block event is "latest," producing cross-family snapshot incoherence for the same key. Fix: one shared sort-key helper.
14. **[minor/low] cross-chain ordering by raw block number in the ENS branch of `name_current`** — `name_current/load.rs:296-314` has no chain filter (unlike the basenames branch) and orders all events by `block_number`; if a logical name ever has both ENSv1-mainnet and ENSv2-sepolia events (both derivation kinds are included, `load.rs:188-193`), the interleave is meaningless. Profile separation makes co-occurrence unlikely today.

### IMPROVEMENTS

15. **[minor/high] duplicated derive loops** — `run_once` performs a derive pass (`projection_apply.rs:50`) and a separately spawned loop also derives continuously (`automatic_projection_replay.rs:152-157,226-269`). Serialized by `FOR UPDATE` on the cursor row, so safe, but one of the two is redundant polling load.
16. **[minor/high] near-duplicate SQL blocks** — the two `name_current` load branches differ only by a `source_family` filter (`name_current/load.rs:201-324`); `CANONICAL_STATE_FILTER` is re-declared in five family modules (`name_current.rs:95`, `permissions.rs:25`, `record_inventory/constants.rs:44`, `resolver.rs:55`, `address_names/constants.rs`). Worth consolidating.
17. **[minor/medium] point-rebuild upsert + delete-stale are not transactional** — `children.rs:116-118`, `record_inventory/projection.rs:161-165`, `permissions.rs:122-125` run upsert then stale-delete as separate statements; a reader between the two sees old and new rows for the same key family simultaneously. Wrap in one transaction.

### OPTIMIZATIONS

18. **[moderate/medium] claim query cannot use the pending index** — `projection_invalidations_pending_idx` is `(projection, last_changed_at, projection_key) WHERE claim_token IS NULL` (`migrations/20260507120000_projection_apply_queue.sql`), but the claim's `OR claimed_at < now()-…` predicate and `ORDER BY CASE …` (`apply.rs:290-316`) force a scan-and-sort of the full pending set per 25-row claim. During catch-up (the doc's "no apply cursor → derive from the beginning" path, projections.md:228) the pending set can be millions of rows, making every iteration O(n log n).
19. **[minor/high] children full-rebuild block cache is per-task and therefore useless** — `spawn_children_rebuild_task` creates a fresh `BTreeMap` per single-row task (`children.rs:98-108`), so the `CHILDREN_CURRENT_BLOCK_CACHE_LIMIT=4096` cache never gets a hit in the full-rebuild path; every child row does its own `load_raw_block` round-trip. The single-parent path shares the cache correctly (`children.rs:127-139`). Share a bounded cache across tasks or batch rows per parent.
20. **[minor/medium] row-at-a-time applies with RPC inside the claim** — apply handles one invalidation at a time (except the address grouping, `apply.rs:53-73`), and `record_inventory` applies run Multicall3 text hydration inline while holding the claim and the global apply/hydration mutex (`apply.rs:372-390`, lock at `automatic_projection_replay.rs:200`), stretching claim durations (compounds findings 10/11). Hydration could be deferred to a post-batch pass.
21. **[minor/low] `resolver_current` full rebuild concurrency is 1** (`resolver.rs:44`) versus 8-32 elsewhere — if intentional (fan-in cost), a comment would help; otherwise an easy win.

### Three biggest themes

1. **Invalidation triggers and projection consumption have drifted apart.** Several documented inputs either trigger rebuilds that ignore them (`PermissionScopeChanged` → fuse masking never applied) or are consumed by rebuilds but never trigger them (ENSv2 `ParentChanged`, manifest version changes). Each derive prefix should be reconciled against its family's actual rebuild inputs, in both directions.
2. **The queue's safety story stops at the queue row.** The generation guard, lease reclaim, and heartbeats protect `projection_invalidations` integrity, but nothing serializes the projection-table writes themselves — so concurrent workers can publish stale rows (claim-clear on upsert, unheartbeated batch tails), and there is no poison handling to stop a permanently failing key from looping forever and blocking hydration.
3. **Full-rebuild atomicity and marker hygiene are inconsistent across families.** Two families have a correct staged-replacement publish; five wipe the live table first, and the per-family CLI commands skip the documented marker-clearing rule — together this is the one path where a crash leaves silently incomplete projections that bootstrap will never repair.

## 4. crates/adapters

### Overall assessment

The adapter layer is carefully engineered where it has been pointed (migration suppression ordering, deterministic IDs, idempotent event identities, label/namehash cross-checks, checkpointed replay), and the migration-suppression core matches the subgraph semantics. The most serious problems are *coverage* holes in the decode surface rather than wrong math: the ENSv1 registry `Transfer(bytes32,address)` event (setOwner) is suppression-checked but never decoded anywhere, NameWrapper `TransferBatch` is unwatched, and ENSv2 registry-side `EACRolesChanged` is unconsumed. The registry-`Transfer` hole then compounds with this branch's new "registry-owner supersedes registrar" heuristic to break the documented unwrap-reactivates-prior-lease invariant. Several taxonomy event kinds are never emitted by any adapter, and one undocumented kind is.

### GAPS

1. **[significant/high]** `crates/adapters/src/ens_v1_unwrapped_authority/observation.rs:6-383` — ENSv1/Basenames registry `Transfer(bytes32,address)` is never decoded into any observation. The signature is selected (`constants.rs:58`, `event_topics.rs:21-26`) and suppression-checked (`migration_guard.rs:54-57`), and subregistry discovery also only builds NewOwner/NewResolver assignments (`ens_v1_subregistry_discovery/assignment.rs:144-151`), but no `RegistryOwnerChanged` is ever produced from it — confirmed crate-wide. Upstream `setOwner` emits only `Transfer` (`.refs/ens_v1/contracts/registry/ENSRegistry.sol:63-69`) and the reference indexer updates owners from it (`.refs/ens_subgraph/src/ensRegistry.ts:147-157`). Consequences: every `setOwner`/`setRecord` owner change is invisible (`current_registry_owner` stale, no `AuthorityTransferred`), and NameWrapper's non-.eth `wrap`/`unwrap` paths — which move registry ownership solely via `setOwner` (`.refs/ens_v1/contracts/wrapper/NameWrapper.sol:372`) — leave permanently wrong owner state. Fix: add a `Transfer` decode branch producing `RegistryOwnerChanged { namehash, owner }` (the struct already supports namehash-only targeting).
2. **[significant/high]** `crates/adapters/src/ens_v1_unwrapped_authority/event_topics.rs:50-56` — `WRAPPER_EVENT_SIGNATURES` includes `TransferSingle` but not `TransferBatch(address,address,address,uint256[],uint256[])`. The subgraph handles it (`.refs/ens_subgraph/subgraph.yaml:222-223`). Wrapped-name owner changes via `safeBatchTransferFrom` are silently missed → stale wrapper owner, missing `TokenControlTransferred`/`PermissionChanged`. Fix: add the signature and fan out per-id into `WrapperTokenTransferObservation`.
3. **[moderate/high]** `crates/adapters/src/ens_v2_permissions/load.rs:32`, `ens_v2_registry/constants.rs:37-47` — ENSv2 `EACRolesChanged` is consumed only from the **resolver** family; the PermissionedRegistry/RootRegistry's own role events (the core ENSv2 permission surface, `.refs/ens_v2/contracts/src/access-control/interfaces/IEnhancedAccessControl.sol:24-29`) are watched nowhere. docs/architecture.md:257 maps `PermissionChanged` *and* `RootPermissionChanged` from registry EAC events. Registry-resource and root-resource permissions are simply absent.
4. **[moderate/high]** Taxonomy kinds never emitted by any adapter (confirmed by crate-wide search): `RootPermissionChanged` (root resources get plain `PermissionChanged` with `root_resource:true`, `ens_v2_permissions/normalized.rs:64-150`), `RecordDeleted` (`DNSRecordDeleted` is mapped to `RecordChanged` with `{"deleted":true}` — `ens_v1_unwrapped_authority/observation/resolver_records.rs:158-173`), `CommitmentMade` (upstream emits it, `.refs/ens_v2/contracts/src/registrar/interfaces/IETHRegistrar.sol:19`; `ens_v2_registrar.rs:31-34` only decodes NameRegistered/NameRenewed), and `DelegateRetainedAfterTransfer`. Conversely `ens_v2_registrar.rs:29` emits `RegistrarNameRegistered`, which is not in the documented taxonomy (architecture.md:235-249).
5. **[moderate/high]** `crates/adapters/src/block_derived_normalized_events/constants.rs:21-24,41-44` — `PreimageObserved` intake for the ENSv1 registrar family covers only the legacy controller signatures (5-field `NameRegistered`, 4-field `NameRenewed`). The wrapped (6-field) and unwrapped (7-field + referrer) controller variants — admitted in the same family and decoded by the authority adapter (`ens_v1_unwrapped_authority/constants.rs:27-37`) — and both Basenames controller events produce no preimage observations, so post-2023 registrations never feed `label_preimages` from this path.
6. **[moderate/medium]** `crates/adapters/src/ens_v2_registry/constants.rs:37-47` — the ENSv2 registry's ERC-1155 `TransferSingle`/`TransferBatch` (post-registration ownership transfers) are unwatched; owner state set at `LabelRegistered` never updates and v2 emits no `TokenControlTransferred`/`AuthorityTransferred` for transfers. (sepolia-dev scope softens severity.)
7. **[minor/high]** `crates/adapters/src/ens_v1_unwrapped_authority/observation.rs:20-58` — registrations whose label fails ENSIP-15 normalization, or whose *normalized* labelhash ≠ indexed labelhash (e.g. `"Alice"` → labelhash mismatch), are dropped entirely: no lease, no expiry, no release, no `TokenTransferred` (transfers early-return when no registration exists, `apply_registrar.rs:344-360`). The subgraph retains these under bracketed labelhashes. The doc's bracketed-placeholder mechanism covers children rows but not registrar authority.
8. **[minor/medium]** `crates/adapters/src/ens_v1_unwrapped_authority/event_topics.rs:43-49` — Basenames resolver intake omits `ContenthashChanged`/ABI/DNS profiles that the Base `L2Resolver` exposes; and pre-2019 `.eth` migration mints (`NameMigrated`, no controller label event) are invisible until first renewal. Both appear to be the documented "label-bearing only / capability narrowing" — listing here for completeness since neither is in upstream.md's divergence list.

### INCORRECT BEHAVIOR

1. **[significant/high]** Unwrap does not reactivate the prior registrar lease (violates architecture.md:88-96 row "Unwrap before lease ends → reactivate prior registrar"). At `wrapETH2LD` the ERC721 `Transfer(user→wrapper)` (`.refs/ens_v1/contracts/wrapper/NameWrapper.sol:264-268`) sets `lease.registrant = wrapper` while `current_registry_owner` is still the user, so `registry_owner_still_supersedes_registrar` fires and demotes the live lease to `superseded` (`apply_registrar.rs:407-427, 568-575`). At `unwrapETH2LD` the registry handover is a `setOwner` → `Transfer` event (`NameWrapper.sol:382-396` via `_unwrap`) which is never decoded (Gap 1), so when the ERC721 `Transfer(wrapper→user)` arrives, the restore condition `current_registry_owner == to` (`apply_registrar.rs:443-447`) compares the stale wrapper address and fails. Result: after a standard wrap→unwrap round trip the surface binds to the node-scoped registry-only resource (owner recorded as the NameWrapper address) instead of reactivating the original registrar resource/token lineage — and later renewals keep it superseded (`restore_superseded_registration_for_renewal`, `apply_registrar.rs:486-489`).
2. **[significant/medium]** Stale wrapper authority across lapse + re-registration. Nothing ever clears `current_wrapper_key` except an observed `NameUnwrapped`: `settle_due_registration_release` early-returns when wrapped (`apply_registrar.rs:303-305`) and `apply_registration_granted` doesn't touch it. A wrapped name that expires past grace and is re-registered through the unwrapped 2024 controller keeps the dead wrapper anchor as the active binding (`transition.rs:235-243` prefers wrapper), `transition_authority(wrapper→wrapper)` no-ops, and the new lease never becomes the bound resource — no `AuthorityEpochChanged`, wrong resource continuity. Fix: expire/clear the wrapper authority when the underlying .eth lease releases or a fresh `RegistrationGranted` arrives without a same-tx `NameWrapped`.
3. **[moderate/medium]** Registry-owner-divergence rotation contradicts the docs. `apply_registry_owner_changed` (`apply_registry.rs:9-21`) and the transfer path demote a live lease and rebind the surface to a registry-only resource whenever registry owner ≠ registrant. architecture.md:60 says the same `resource_id` "persists across holder, controller, resolver, expiry, grace, fuse, and status changes", and the rebind table (:88-96) lists no owner-divergence case; upstream the registrant retains `reclaim` authority for the whole lease (BaseRegistrarImplementation.sol:172-175). This branch's behavior may be intended ("fix: repair ENSv1 registry-owner authority") but is not reflected in docs or upstream.md § Known divergences — either document it or restrict rotation to genuine anchor moves.
4. **[minor/high]** Grace-boundary off-by-one. Release is computed as the first block with timestamp ≥ expiry+90d and treated as released when `release_ref.block_timestamp <= ts` (`transition.rs:287-294`, `apply_registrar.rs:577-585`), but upstream `renew` is valid while `expiries + GRACE >= block.timestamp` (`.refs/ens_v1/contracts/ethregistrar/BaseRegistrarImplementation.sol:161`) and availability needs strict `<` (:101-104). A renewal in a block whose timestamp exactly equals expiry+grace is processed after `settle_due_registration_release` already released the lease (`apply.rs:8-11`) → spurious `RegistrationReleased` plus a brand-new resource/lineage instead of a renewal. Release should require `block.timestamp > expiry + grace`.
5. **[minor/medium]** Zero-length binding segments vanish silently. `transition_authority` (`transition.rs:16-57`) drops an open binding closed at the same timestamp without pushing a segment or emitting `SurfaceUnbound`, while the earlier `SurfaceBound` normalized event survives — register+wrap in one tx leaves a dangling `SurfaceBound` whose `surface_binding_id` never materializes. Out-of-order anchors (`active_from > effective_time`) are likewise discarded with no trace.
6. **[minor/high]** Broken before_state values: `RegistrationGranted.before_state.registrant` is always JSON null because of `.and_then(|v| v.token_lineage_id).map(|_| Value::Null)` (`apply_registrar.rs:54`) — the prior registrant is lost; the renewal-synthesized `RegistrationGranted` carries an empty before_state (`apply_registrar.rs:181`).
7. **[minor/medium]** `apply_record_changed` drops record events whose emitting resolver isn't the node's *current* resolver (`apply_resolver.rs:109-111`). The common migration flow "write records to new resolver, then `setResolver`" loses the final record state (no replay after `NewResolver`); the subgraph keeps per-(resolver,node) state. Generic observed-selector intake doesn't recover declared `RecordChanged` history for the new binding.
8. **[minor/medium]** `crates/adapters/src/registry_migration_cache.rs:131-166` — the process-global migrated-nodes cache grows monotonically and never invalidates entries whose marking `NewOwner` was orphaned by a reorg; old-registry events for such nodes stay suppressed until process restart. (Order-correctness for live intake is otherwise fine: markers are applied in (block, tx, log) order — `loading/raw_logs.rs:94-101, 406-413`.)

### IMPROVEMENTS

- **[minor/high]** Two diverging migration-guard implementations: `ens_v1_subregistry_discovery/migration_guard.rs:58-65` has an explicit `OldRootResolverException` (ZERO node), while `ens_v1_unwrapped_authority/migration_guard.rs:49-52` relies on the implicit fact that the root node can never be a `NewOwner` child. Behavior matches the subgraph (`.refs/ens_subgraph/src/ensRegistry.ts:243-249`) in both, but the implicit variant is fragile and the duplication invites drift — share one guard.
- **[minor/medium]** `ens_v1_reverse_claim/events.rs:61-71` — a `ReverseClaimed` whose indexed node mismatches the derived node `bail!`s the whole sync batch instead of dropping/flagging one log; same brittleness for malformed `AliasChanged` in `block_derived_normalized_events/event_builders.rs:199-210`.
- **[minor/medium]** Registrar label decoding + ENSIP-15 observation logic is duplicated between `ens_v1_unwrapped_authority/{abi.rs,names.rs}` and `block_derived_normalized_events/{event_builders/label_preimages.rs,preimage_observation.rs}` — the signature sets have already diverged (Gap 5); `can_observe_dns_label` also lost the NUL check its twin has (`names.rs:8-13` vs `preimage_observation.rs:9-11`).
- **[minor/high]** `finalization.rs:113-117` — `anchor.binding_manifest_id.checked_mul(0).unwrap_or_default() + ts` is an obfuscated no-op; also the registry-only fallback binding opens at the batch head timestamp rather than the owning observation's timestamp (`finalization.rs:109-122`).
- **[minor/medium]** Silent drops with no observed-only trace: ENSv2 `TokenRegenerated`/`TokenResource` for unknown tokens (`ens_v2_registry/events.rs:386-462`), unnormalizable `LabelRegistered` names (`events.rs:54-57`), and `NewTTL` (selected, suppression-checked, never built).

### OPTIMIZATIONS

- **[moderate/high]** `ens_v2_registrar/resource_links.rs:13-56` — one JSONB-expression query against `normalized_events` (`after_state ->> 'token_id'` etc., unindexable as written) **per matched registrar log**; batch per sync or maintain a token→resource side table.
- **[minor/medium]** `ens_v1_unwrapped_authority/preload.rs:66-242` — ~8 sequential preload queries per restricted batch (registrar state, selected registrar, wrapper, registry owner, resolver ×3, record versions); these are per-block in live intake and could be combined or cached across consecutive blocks.
- **[minor/medium]** `loading/raw_logs.rs:127-149` — `topics`/`data` are cloned once per source-candidate per row in the streaming path; an `Arc` or restructuring to per-row candidates-after-build would cut hot-path allocations on the 100k-logs/page replay.
- **[minor/low]** `registry_migration_cache.rs:139-165` — global `Mutex` + full `HashSet` clone per load call; an `Arc<HashSet>` swap would avoid copying the (large, mainnet-scale) migrated-node set on every restricted sync.

### Three biggest themes

1. **Decode-surface holes, not decode bugs** — registry `Transfer`, wrapper `TransferBatch`, ENSv2 registry `EACRolesChanged` and ERC-1155 transfers, and post-legacy controller preimage signatures are all selected-or-adjacent events that never become observations, and they account for most of the wrong-state risk.
2. **The wrap/unwrap ↔ registry-owner heuristic is the fragile core of ENSv1 resource continuity** — the new supersede logic interacts badly with undecoded `setOwner` events and timestamp-equal boundaries, breaking the documented unwrap-reactivation and lapse-rotation invariants exactly where the docs are most explicit.
3. **Taxonomy drift** — several documented event kinds are never produced and one undocumented kind is, plus duplicated guard/normalization logic across modules has already diverged; tightening the kind constants and sharing the ENSv1 decode/migration-guard primitives would prevent both classes of drift.

## 5. crates/storage + migrations

### Overall assessment

This is an unusually disciplined persistence layer: canonicality is explicit everywhere, upserts are conflict-target-correct with atomic SQL-side merges on the bulk paths, repair gates are narrow and fail-closed by design, and the identity-facade sidecars (counts + feed) converge on a single readable-universe definition that the API read path mirrors exactly (`identity_facade/mod.rs` filters match the SQL functions in migration `20260604120000`). The main problems are: one structural SQL bug in the largest new repair module (validation placed in a `LEFT JOIN ... ON` clause where it cannot filter), several documented invariants the schema/migrations don't actually honor (partitioning, projection-row metadata, interval tighten-only), missing indexes on the reorg-repair write path, and trigger/over-invalidation overhead on hot write paths that the team has been chasing incrementally (five "skip noop" migrations) without closing the loop on promotions and the feed trigger.

### GAPS

- **[significant/high] migrations/20260430060000_baseline.sql:227,673,875,959,998 — documented partitioning baseline absent.** docs/storage.md:191–203 declares `chain_lineage`, `chain_header_audit`, `raw_transactions`, `raw_receipts`, `raw_logs`, `normalized_events`, `execution_steps` partitioned by `chain_id` + block-number range. Every one is a plain `CREATE TABLE`; no later migration adds partitioning. Either partition (painful now: PK/unique constraints would need the partition key) or fix the doc — as-is the doc misleads capacity planning and explains several seq-scan findings below.
- **[moderate/high] crates/storage/src/normalized_events/orphaning.rs:33–46 and crates/storage/src/identity/orphan.rs:156–183 — reorg orphaning updates have no supporting index.** Both filter `chain_id = $1 AND block_hash = ANY($2)`. `normalized_events` has no `(chain_id, block_hash)` index (all baseline indexes lead with `logical_name_id`/`resource_id`/`block_number`; verified across all migrations), nor do `name_surfaces`, `resources`, `surface_bindings`, `token_lineages`. Reorg repair — the most latency-critical write path — degenerates to per-chain scans of the largest tables. The `raw_*` tables are fine (unique `(chain_id, block_hash, …)` constraints). Add `(chain_id, block_hash)` btrees, or constrain by `block_number` range (already known from the lineage walk).
- **[moderate/high] migrations/20260430060000_baseline.sql:786–801 — `primary_names_current` lacks the documented projection-row columns.** docs/storage.md:316 ("Every current-state projection row carries provenance pointers, manifest version, relevant chain positions, canonicality summary, and last-recomputed timestamp") — the table has only `claim_provenance` jsonb (positions are buried inside it; see `crates/storage/src/primary_name/hooks.rs`), no `manifest_version`, `chain_positions`, `canonicality_summary`, or `last_recomputed_at`. Snapshot-eligibility checks for this projection can't be done structurally.
- **[moderate/medium] migrations/20260502170000_app_facing_rest_indexes.sql:3 + crates/storage/src/name_current/list.rs:428–444 — `pg_trgm` installed, never used.** The list path's `contains`/`contains_nocase` (`LIKE '%…%'`) and the resolver filter (`LOWER(declared_summary #>> '{resolver,address}') = $n`) have no supporting GIN-trgm/expression index; the only `name_current` name indexes are plain btrees, so these documented filters are full scans.
- **[minor/high] migrations/20260507120000_projection_apply_queue.sql:1–13 — `projection_normalized_event_changes` has no pruning path.** One row per normalized-event insert plus one per canonicality update (including mass reorg orphan updates and every repair). No `DELETE FROM projection_normalized_event_changes` exists anywhere in the repo; the cursor table only tracks watermarks. Unbounded append-only growth on the hottest trigger.

### INCORRECT BEHAVIOR

- **[significant/high] crates/storage/src/normalized_events/upsert/repair/ens_v1_registry_event_time.rs:208–633 — resource-anchor validation lives in a `LEFT JOIN ... ON` clause, so it never filters.** The entire validation block — new-resource canonicality/logical-name/labelhash checks (211–316), the same-transaction `RegistrationGranted` proof (260–291), the nullable-resource gate (295–303), and the per-kind before/after-state proofs that reference `new_resource.provenance` (317–633) — is attached to the `LEFT JOIN resources new_resource` `ON` condition. In a LEFT JOIN, ON-clause predicates never remove left-side rows; they only decide whether `new_resource` columns are null, and `repair_map` selects only `input.*`. Consequently every conflicted event that passes the (anchor-blind) Rust shape gate `ens_v1_unwrapped_authority_registry_event_time_resource_id_repair_allowed` (:1088) and the `old_resource` INNER JOIN (:187–207) is repointed by the `updated` CTE (:636–646) to whatever `new_resource_id` the adapter supplied — orphaned, wrong-name, wrong-labelhash, or null — and the fail-closed bail at :822–827 ("rejected invalid resource anchors") cannot fire for those classes. This contradicts docs/storage.md:296 ("Repaired registry-only resources must be canonical mainnet resources for the same logical name and leftmost labelhash…"). The sibling modules do this correctly (`ens_v1_renewal.rs` builds `resource_candidates` with INNER JOINs; `ens_v1_registry_event_time_null_resource.rs:98–120` uses a `WHERE`), and no test exercises SQL-level rejection for this path (the only anchor-rejection test is the renewal one, `normalized_events/tests.rs:4788`). Fix: move the validation into a `WHERE` over the LEFT JOIN result.
- **[moderate/medium] crates/storage/src/identity/write_rows/surface_binding.rs:71–117 (same pattern in write_rows.rs:66–137, 184–259, 358–447) — read-modify-write without `FOR UPDATE`.** On insert conflict the existing row is reloaded with a plain SELECT (`identity/read.rs:212`), merged in Rust (`merge_binding_active_to`, `merge_observation`), then blindly `UPDATE … SET active_to=$2, canonicality_state=$3`. A concurrent tighten (or reorg-repair `orphaned`) committed between read and write is overwritten — re-extending `active_to` or reviving an orphaned row, violating the tighten-only/orphan rules in docs/storage.md:111. The bulk path proves the merge atomically in SQL (`write_fast.rs:736–753` uses `LEAST(...)` and the canonicality CASE against the live row). Single-writer deployment makes this latent, not impossible.
- **[moderate/medium] migrations/20260508203000:333–345, migrations/20260514110000 (same block), migrations/20260508204000:268–281 — repairs extend or reopen closed binding intervals.** 203000/0514110000 set the expected binding's `active_to` to the *later* old-binding close (`binding.active_to < old_binding.active_to` → extend); 204000 copies `old_binding.active_to` even when it is NULL (reopening a closed interval; its second OR arm at :277–280 is also redundant with `IS DISTINCT FROM`). docs/storage.md:111 says replay "does not extend or reopen a closed interval" — these are repair migrations, but the doc carves no repair exception, and an extension overlapping a successor canonical binding aborts on `surface_bindings_no_overlap` (baseline.sql:1506). At minimum the doc needs the repair carve-out.
- **[moderate/medium] migrations/20260508200000:70–78,164–175, 20260508201000:102–110,194–202, 20260514110000 — `RegistrationReleased` identity rewrite without collision handling.** `event_identity = replace(identity, old_key, new_key)` with no `NOT EXISTS` guard; if the repaired identity already exists the unique constraint (baseline.sql:1346) aborts the whole migration. The Rust renewal repair handles exactly this (`ens_v1_renewal.rs` `release_identity_collisions` CTE orphans the stale row, as docs/storage.md:293 requires); 20260508208000:293–302 also guards. The three migrations don't.
- **[minor/high] crates/storage/src/history/paging.rs:159–170 — same-block history ordering uses `transaction_hash DESC` before `log_index DESC`.** `log_index` is block-scoped in Ethereum, so ordering by transaction hash first renders multi-transaction blocks in hash order, not execution order. Deterministic but non-chronological; the baseline replay indexes encode the same column order (baseline.sql:1788, 1816). Projection rebuild queries order by `log_index` directly, so this is a history-read presentation issue only.
- **[minor/high] crates/storage/src/identity/write_rows.rs:104–135, 221–257, 395–445 — conflict refresh always rewrites `observed_at = now()`,** even for byte-identical re-observations. docs/storage.md:109 says compatible replays may be accepted "without rewriting the original anchor, anchor provenance, or `observed_at`"; the bulk paths honor this (`write_fast.rs:189,417`), the row paths don't — doc divergence plus dead-tuple churn and needless statement-trigger executions.

### IMPROVEMENTS

- **[moderate/high] Repair-migration proliferation vs the generic mechanism.** Fourteen ENSv1 repair migrations (`20260508130000` → `20260514180000`) share a near-identical ~150-line 4×`CROSS JOIN LATERAL jsonb_set` grant/revocation-key rewrite block plus the same orphan/queue tail; later ones bootstrap off `_sqlx_migrations.installed_on` (`20260514110000:14–22`) and wall-clock `observed_at >= '2026-05-14'` cutoffs (`20260514120000:127`) — fragile coupling to migration history. The Rust repair framework under `crates/storage/src/normalized_events/upsert/repair/` is the right generic mechanism and now duplicates most of this logic; future repairs should land only there (or as shared SQL functions), with migrations reduced to invoking them.
- **[moderate/high] Readable-universe predicate restated 6+ ways.** `identity_facade/mod.rs:24–80` (Rust constants, inlined into 4 UNION branches in `reverse_page.rs`), the SQL functions in `20260604120000` (`…_row_readable`, `…_readable_relation_rows`, feed candidates), and the count/feed rebuild CTEs all restate the same canonical+open-binding+reachable-`name_current` join. The SQL function `address_names_current_identity_readable_relation_rows` already exists — the API read path should call it (or a view) instead of re-inlining, so the next readability change can't drift.
- **[minor/medium] `change_kind='canonicality_update'` is overloaded for payload/resource repairs** — every repair migration (e.g. `20260508130000:118–131`) and Rust repair CTE (`upsert/repair.rs:318–335`) records repairs as `canonicality_update` even when canonicality didn't change. Consumers can't distinguish "payload rewritten" from "state promoted"; a third `change_kind` would make the change log honest (the CHECK in `20260507120000:7–9` would need extending).
- **[minor/medium] migrations/20260604123000:174–178 — the "repair" invalidation trigger fires on every `surface_bindings` UPDATE statement,** including routine live closes already covered by normalized-event changes, double-enqueueing `name_current`/`address_names_current` keys. Harmless (idempotent queue) but contradicts the migration's repair-only framing and adds queue churn on the hot path.

### OPTIMIZATIONS

- **[significant/medium] Canonicality-promotion trigger storms.** The statement triggers in `20260604131000:383–462` and the invalidation trigger in `20260604123000:17–26` fire on any `canonicality_state` change — including `canonical→safe→finalized` promotion, which is read-equivalent in every readability filter. A finalized backfill replay over existing rows (the `merge_observation` promotion path, `write_fast/sql.rs:1–40`) bulk-promotes identity rows and triggers full per-address count+feed recomputes plus mass `projection_invalidations` for zero semantic change. Gate the triggers/filters on readability-class transitions (`orphaned/observed ↔ canonical|safe|finalized`) instead of raw inequality.
- **[moderate/high] migrations/20260521160000:476–479 — `address_names_current_identity_feed_after_change` is row-level `AFTER INSERT OR UPDATE OR DELETE` with no `OF`/`WHEN`,** and the projection upsert always rewrites the row (`address_names/write.rs:158–173`, `last_recomputed_at` always changes). Every anc refresh therefore runs a full per-address feed recompute (delete + 5-table aggregate). The counts trigger was narrowed in `20260520172000:352–356`; the feed trigger never was — at minimum add `WHEN (OLD.* IS DISTINCT FROM NEW.*)` or an `OF` column list covering displayed fields.
- **[moderate/high] migrations/20260608160000_label_preimage_invalidation_lookup.sql:1 — non-concurrent `CREATE INDEX` on `normalized_events`,** the largest table, with no `-- no-transaction` marker, unlike its siblings `20260603182000/182100` which use `CONCURRENTLY`. Blocks all event writes for the build on a production-sized table. Same (lower-stakes) pattern on projection tables in `20260502170000`, `20260519190000`, `20260521143000`.
- **[moderate/medium] More redundant indexes beyond the 20260521170000 cleanup:** `normalized_events_primary_names_claim_lookup_idx` (`20260508160000:27–50`) is fully subsumed by `…claim_prepared_lookup_idx` (`20260508170000`, identical keys, strictly weaker predicate that's the only one provable for prepared statements); `discovery_edges_active_source_to_endpoint_idx` (baseline.sql:1649) is a strict prefix of `discovery_edges_active_source_target_kind_chain_idx` (`20260514100000`) under the same predicate; `discovery_edges_active_transitive_parent_scope_idx` duplicates `…active_to_endpoint_scope_idx` keys with a contained predicate; `normalized_events_history_resource_lookup_idx` (`20260503100000`) is nearly a prefix of the baseline `normalized_events_resource_projection_replay_idx`.
- **[moderate/medium] crates/storage/src/execution/invalidation.rs:176–221 — reorg cache invalidation loads every verified outcome and every orphaned lineage row into memory per call** and matches dependencies in Rust; O(total cache) per reorg and the orphaned-block set only grows. Push the dependency match into SQL or scope by recently-orphaned blocks.
- **[minor/high] crates/storage/src/normalized_events/upsert.rs:170–174, 349–353 — per-event loop over `upsert_normalized_event_batch`** (which accepts a slice) for canonicality refreshes: N round trips where one batched statement suffices.

### Three biggest themes

1. **Validation that looks enforced but isn't.** The flagship example is the `LEFT JOIN ON`-clause placement in `ens_v1_registry_event_time.rs`, which silently disables all resource-anchor proofs the docs require and the bail message claims; the missing rejection tests for that path are the second half of the same theme. SQL-side gates in fail-closed repair paths need either INNER-JOIN/WHERE structure or explicit rejection tests.
2. **Repair grew as ~20 one-off migrations before the generic mechanism existed,** leaving duplicated jsonb-rewrite boilerplate, wall-clock/migration-timestamp coupling, doc-invariant violations (interval extension/reopen), and unguarded identity rewrites in the migration chain — while the newer Rust repair framework does the same jobs more safely. Future repairs should be code-first.
3. **Trigger-based sidecars are correct but pay for every write.** The counts/feed/invalidation triggers converged on the right readable-universe semantics, but they still fire on read-equivalent canonicality promotions, on every `address_names_current` rewrite, and append an unprunable change-log row per event — the cumulative write amplification on replay/backfill is the storage layer's largest unaddressed cost, and the docs' promised partitioning was never implemented.

## 6. crates/execution + crates/manifests + manifests/

### Overall assessment

Both crates are carefully written, heavily validated, and unusually disciplined about fail-closed persistence: the resolution persistence path revalidates topology/support class from storage before committing (crates/execution/src/revalidation.rs:25-117), cache keys carry block-hash-bearing dependencies (crates/storage/src/execution/keying.rs:135-173), and reorg invalidation is genuinely wired into the indexer's repair paths (apps/indexer/src/main/reconciliation/canonical/orphaning.rs:67, apps/indexer/src/main/rewind.rs:129). ENSIP-10 wildcard traversal is delegated to the on-chain Universal Resolver and the local class derivation faithfully implements the documented three ENS classes plus the Basenames transport class (crates/storage/src/resolution_support/topology.rs:18-136). The checked-in TOML manifests are internally consistent and match docs/architecture.md addresses and start_blocks exactly. The main problems are on the edges the docs promise but the code doesn't close: verified-primary execution has no production producer and no automatic claim-change/manifest-change invalidation, the CCIP-Read client deviates from EIP-3668 in URL-template and retry handling, the RPC client refuses `https://` endpoints, and the error taxonomy collapses nearly every failure into `resolver_call_failed`.

### GAPS

- **[significant/high] crates/execution/src/persistence.rs:193 + crates/storage/src/primary_name/hooks.rs:12 — verified-primary execution has no production producer and the claim-change invalidation hook is dead code.** `persist_ens_verified_primary_name` is exported but called only from tests (no caller in apps/), so `request_type=verified_primary_name` rows can only appear via tests or out-of-band backfill. Meanwhile the worker projection writes `verified_primary_name_invalidation` hooks into `primary_names_current.claim_provenance` (apps/worker/src/primary_name/projection.rs:204-236) and storage ships a decoder (`verified_primary_name_claim_hooks`), but nothing in production consumes it — the only invalidation paths are manual worker CLI commands (apps/worker/src/commands/execution_invalidation.rs:9-24). Docs require invalidation on "primary claim change" (docs/execution.md:134-142, docs/architecture.md:417). Fix: wire the projection's claim-change path to `invalidate_execution_outcomes_for_*_and_request_key`, and either add the production verified-primary executor or document the class as backfill-fed.
- **[significant/high] apps/api/src/support/primary_name_lookup.rs:78-104 — verified-primary readback keys on `request_key` only, so stale verified answers survive claim changes.** The API selects the newest `execution_cache_outcomes` row by `(request_type, namespace, request_key)` ordered by `finished_at DESC`, ignoring `requested_chain_positions`, `manifest_versions`, and both version boundaries; the only claim-side gate is row existence (crates/execution/src/persistence.rs:258-268 has the same shape). Docs say the cache is keyed by all five components and that the matching `primary_names_current` row is the invalidation anchor (docs/execution.md:122-132). Combined with the missing claim-change invalidation above, a user who changes or clears their reverse claim keeps being served the old `verified_primary_name.status=success` payload. Fix: compare the cached outcome's boundaries against the current claim row (or filter by full cache key) before serving readback.
- **[significant/high] crates/execution/src/rpc.rs:80-84 — `https://` RPC endpoints are rejected.** `JsonRpcHttpClient::new` bails on any scheme other than `http`, so every on-demand execution path (verified resolution, ENS/60 primary fallback, reverse-name and text-record multicalls) requires a plaintext `http://` provider URL. Docs only say "the API Ethereum RPC provider must be configured" (docs/execution.md:36) with no scheme restriction; virtually all hosted providers are https-only. Fix: accept https (the comment admits this is a bootstrap shortcut).
- **[moderate/medium] docs/manifests.md:232-236 vs apps/worker/src/commands/execution_invalidation.rs — manifest-change cache invalidation is operator-driven, not event-driven.** Manifest sync produces `SourceManifestUpdated`/`CapabilityChanged`/`ProxyImplementationChanged` normalized events (crates/adapters/src/manifest_normalized_events/constants.rs:3-5), but nothing consumes them to "invalidate execution cache entries" as documented — the invalidation functions exist only behind worker CLI subcommands. Resolution readback is partially protected because the lookup re-derives the full cache key from current projection state (apps/api/src/responses/resolution_verified/readback.rs:38-56), but verified-primary readback (above) is not. Fix: trigger `invalidate_execution_outcomes_for_manifest_version` from the manifest-event sync path.
- **[moderate/medium] crates/manifests — documented discovery edge kinds are never produced.** Code produces: authored `subregistry` and `resolver` rules (all checked-in manifests), managed `proxy_implementation` and `migration` edges (crates/manifests/src/lib.rs:29-34, lib/managed_edges/source_graph.rs:19-49). The docs' edge vocabulary also includes `parent`, `alias`, `metadata`, and `transport` (docs/architecture.md:180), and docs/manifests.md:230 says "Manifest-declared and discovered proxy/implementation links share the same edge and active-range rules" — but there is no mechanism to admit a *discovered* proxy/implementation edge (authored `admission` is restricted to `reachable_from_root`, and no rule kind produces one). ENSv2 discovered `UserRegistry` proxies therefore have no implementation linkage even though docs/manifests.md:176 calls `UserRegistryImpl` "implementation metadata".
- **[moderate/medium] manifests/mainnet/ethereum/ens/ens_execution/v1.toml:16-18 — the ENS Universal Resolver proxy is declared `proxy_kind = "none"`.** Docs repeatedly call `0xeEeEEEeE…EeEe` "the official ENS Universal Resolver proxy" with the pinned `.refs` artifact as the implementation anchor (docs/execution.md:24, docs/manifests.md:137). With `proxy_kind="none"` there is no proxy/implementation edge, so manifest drift / proxy-upgrade detection (docs/manifests.md:236) cannot fire when ENS Labs upgrades the UR implementation — exactly the contract whose behavior anchors every verified answer. Fix: declare the real proxy kind + current implementation.
- **[moderate/medium] apps/worker (absent) — no production Basenames verified-resolution executor.** `persist_basenames_exact_name_verified_resolution_transport_direct` (crates/execution/src/persistence.rs:140) is invoked only by tests; the on-demand executor explicitly rejects Basenames (crates/execution/src/ens_resolution.rs:212-216) and the API cache-miss path returns `409 stale` for non-ENS rows (apps/api/src/handlers/resolution_on_demand.rs:49-54). The docs' Basenames class (docs/execution.md:165-173, docs/manifests.md:195) is thus readback-only with no in-repo producer despite `verified_resolution = "supported"` in basenames_execution/v2.toml:15.
- **[minor/high] crates/manifests/src/lib/repository.rs:83-114 — version-tag/`manifest_version` consistency is unvalidated.** A file `v2.toml` containing `manifest_version = 1` parses fine; `ManifestStorageKey` (lib/support.rs:14-25) keys on `manifest_version` only, so it would silently collide with `v1.toml` and one payload overwrites the other during sync. Duplicate `[[contracts]]` roles inside one manifest are also unchecked (DeclarationKey collision in lib/sync/planning.rs:97-123), and `normalizer_version` is never validated against the pinned `ensip15@ens-normalize-0.1.1` boundary the docs declare global (docs/manifests.md:83).
- **[minor/high] crates/execution/src/ens_resolution.rs:383,417 — step `latency_ms` is always `None`.** The trace schema documents per-step latency (docs/execution.md:103-108) and the worker inspect command surfaces it, but no producer ever measures it.

### INCORRECT BEHAVIOR

- **[moderate/high] apps/api/src/support/primary_name_lookup.rs:386-399 — ENS/60 on-demand fallback classifies unnormalizable reverse claims as `not_found` instead of `invalid_name`.** Docs: "A nonblank raw claim that cannot be normalized surfaces `invalid_name`; blank or whitespace-only is `not_found`" (docs/execution.md:64; architecture.md:101). The fallback's `normalize_inferred_route_name` error branch returns `OnDemandPrimaryNameClaimState::NotFound`, so a garbage on-chain `name()` value is indistinguishable from no claim. (The persisted projection path gets this right — apps/worker/src/primary_name/projection.rs:155-165.)
- **[moderate/medium] crates/execution/src/ens_resolution_ccip.rs:237-249 — EIP-3668 URL-template handling is wrong for `{sender}`-only URLs and POST.** The code uses GET when the URL contains `{data}` **or** `{sender}`; per EIP-3668 GET applies only when `{data}` is present, so a template with only `{sender}` is GET-ed without the request data (the gateway never receives the calldata), instead of POSTing JSON to the sender-substituted URL. The POST branch also never substitutes `{sender}` into the URL. Fix: GET iff `{data}` present; always substitute `{sender}` (and `{data}` where present) before either method.
- **[moderate/medium] crates/execution/src/ens_resolution_ccip.rs:196-229 — gateway retry semantics ignore the 4xx/5xx distinction.** Any failure (including HTTP 400-499, which EIP-3668 says indicates a malformed request that MUST NOT be retried) advances to the next URL up to `MAX_GATEWAY_URLS`. Same loop treats malformed-response decode failures as retryable. Fix: stop on 4xx, retry remaining URLs only on 5xx/transport errors. Relatedly (spec deviation, low impact): the client never verifies `OffchainLookup.sender` equals the contract it called before issuing the callback to `lookup.sender` (ens_resolution_ccip.rs:104-111).
- **[moderate/medium] crates/execution/src/ens_resolution_call.rs:100-117,242-249 — OffchainLookup-required and transient RPC failures are persisted as cache-eligible `execution_failed` outcomes.** On the pinned-block (persisting) path CCIP is never followed, so a resolver that reverts `OffchainLookup` produces `execution_failed`/`resolver_call_failed`, which `persist_ens_exact_name_verified_resolution_direct` accepts and which subsequent requests at the same snapshot serve as a cache hit. Docs say CCIP-participating ENS requests are selector-local `unsupported` (docs/execution.md:163) and that `execution_failed` carries a *typed* `failure_reason` (docs/execution.md:20). The crate even ships `rpc_error_contains_offchain_lookup` (ens_resolution_ccip.rs:316) but only the primary-name path uses it (ens_primary_name.rs:419). A transient provider blip is likewise frozen into a persisted failure for that snapshot/selector set. Fix: classify OffchainLookup reverts as unsupported (or a distinct reason) and don't persist provider-transport failures as cacheable outcomes.
- **[minor/medium] crates/execution/src/json_helpers.rs:79-95 — `coin_type` identity is too fine.** `"060"` and `"60"` are both accepted as distinct strings, producing distinct request keys/cache identities for the same coin type; numeric `0` is rejected (`coin_type > 0`) while string `"0"` is accepted. Fix: canonicalize to decimal (parse + re-format) in one place.
- **[minor/medium] crates/execution/src/ens_resolution.rs:464-475 — `manifest_context` hardcodes `ens_execution` `manifest_version: 1`.** When the row provenance lacks an `ens_execution` entry the trace fabricates `{manifest_version: 1}` regardless of the actually-loaded manifest version, so traces would mis-attribute provenance after an ens_execution v2 ships. The cache key path is independent (storage-derived), limiting blast radius to audit material.
- **[minor/low] crates/execution/src/ens_resolution_ccip.rs:90-139 — the depth-limit `bail!` is unreachable.** Every iteration returns unless it `continue`s, and `continue` requires `redirect_index + 1 < MAX_CCIP_REDIRECTS`; on the last iteration a further nested `OffchainLookup` revert is returned as the final call result (becoming a generic `resolver_call_failed`) rather than a distinct "redirect depth exceeded" outcome. Cosmetic, but the docs-level "nested CCIP depth limit" is thus invisible in the taxonomy.
- **[minor/low-medium] crates/manifests/src/lib/managed_edges/source_graph.rs:386-407 — manifest version bumps deactivate all discovered edges sourced from the now-inactive version.** `deactivate_discovery_edges_without_active_source_manifest` deactivates every `discovery_edges` row whose `source_manifest_id` manifest is no longer active — including chain-observed resolver/subregistry edges admitted under the previous version. Recovery depends on a later reconciliation replay re-admitting them under the new manifest_id; in the interim, watch-plan expansion (crates/manifests/src/lib/views/watched.rs) silently loses those targets. Worth an explicit re-admission step inside `sync_repository`.

### IMPROVEMENTS

- **[moderate/high] Error taxonomy collapses.** `decode_rpc_error` deliberately discards `code`, `message`, and `data` and returns a single `resolver_call_failed` (crates/execution/src/ens_resolution_call.rs:242-249); the trace-level failure payload also hardcodes `resolver_call_failed` even when individual selectors failed with `resolver_return_data_malformed` (ens_resolution.rs:283-299); the API's on-demand verification maps every error — configuration, revert, OffchainLookup, transport — to `execution_failed`/`resolver_call_failed` despite the error type carrying `plain_execution_revert`/`offchain_lookup_required` flags built for exactly this purpose (apps/api/src/support/primary_name_lookup.rs:434-449). Docs promise a *typed* `failure_reason` (docs/execution.md:20).
- **[minor/medium] Trace completeness.** Besides the always-`None` latency: gateway interactions on the latest-tag path are recorded only as a digest + count embedded inside the `call_universal_resolver` step payload (ens_resolution.rs:406-411), never as discrete CCIP step rows, so "CCIP-Read participation appears through persisted step kinds" (docs/execution.md:149) currently has no ENS-side producer; the Basenames validator requires such step kinds (validation/path/basenames.rs:64-84) but nothing in-repo emits them.
- **[minor/high] crates/manifests/src/lib/views/watched.rs:16-406 — three near-identical ~130-line SQL bodies with hard-coded source-family mappings.** The resolver-edge → owning-family mapping (`ens_v1_registry_l1`→`ens_v1_resolver_l1`, etc.) is duplicated three times in SQL; adding a namespace requires editing all three. Extract a shared CTE/view or drive the mapping from manifest data.

### OPTIMIZATIONS

- **[moderate/high] crates/execution/src/ens_resolution.rs:247-258 — one `eth_call` per selector.** N requested selectors produce N Universal Resolver `resolve(bytes,bytes)` calls (concurrent, but N RPC round-trips and N raw snapshots). The UR exposes a batched resolve (IUniversalResolver.sol:L52) and the crate already has Multicall3 plumbing (ens_text_records.rs:16-126); the profile route's "every server-derived profile selector" set makes this multiplicative.
- **[minor/high] crates/execution/src/ens_primary_name.rs:181-205 — reverse lookup is two sequential unpinned `eth_call`s** (registry `resolver()`, then resolver `name()`), each at its own "latest"; `execute_ens_reverse_name_multicall` already exists to batch exactly this shape.
- **[minor/medium] crates/storage/src/execution/invalidation.rs:56-100 — manifest/boundary invalidation loads every outcome for the `(request_type, namespace)` scope and filters in Rust;** the JSONB containment checks could be pushed into SQL to avoid full-scope scans.
- **[minor/low] crates/execution/src/rpc.rs:87 and ens_resolution.rs:157 — a fresh `reqwest::Client`/transport per request;** the CCIP path already uses a shared `LazyLock` client (ens_resolution_ccip.rs:17), the RPC path could too.

**Manifest TOML consistency (verified-good):** all addresses, start_blocks, and rollout statuses in `manifests/mainnet` and `manifests/sepolia` match docs/manifests.md exactly (registry 9380380/3327417, registrar 9380410 + controllers 9380471/16925618/22764821, wrapper 16925608, reverse 16925606, resolver generations including the 0xDaaF conservative 9380380 bootstrap, ENSv2 sepolia 10462881/10462895/10462909); exactly one active version per family with older versions `deprecated`; Basenames/UR targets correctly omit `start_block` per the docs' "unknown" rule.

### Three biggest themes

1. **The verified-primary path is half-built relative to its documentation** — no production producer, readback keyed only by request key, and the claim-change/manifest-change invalidation the docs promise exists only as dead hooks and manual CLI commands; this is the one place stale verified answers can actually be served.
2. **The CCIP-Read client cuts corners against EIP-3668** — URL-template method selection, sender substitution, 4xx retry semantics, and sender verification all deviate, and OffchainLookup-required failures leak into the persisted cache as generic failures instead of the documented `unsupported`.
3. **Failures collapse into one bucket** — `resolver_call_failed` covers provider outages, reverts, malformed data, and CCIP requirements alike, while flags and helpers that could type these failures already exist but go unused; together with never-recorded latency, the audit-trail promises of the trace schema are only partially honored.

## 7. Cross-cutting: tests, docs drift, debt, ops, small crates

### Overall assessment

This is an unusually disciplined codebase for its size (~280k lines Rust, 1,184 unit/integration tests, zero TODO/FIXME/unimplemented in production code, docs that mostly verify line-for-line against schema and handlers). The weaknesses are concentrated, not diffuse: the conformance harness — the project's main consumer-contract evidence — is mostly not wired into CI; a handful of doc claims describe load-bearing behavior (wrapper fuse masking, table partitioning, object storage) that does not exist in code; and the deploy path can publish a `latest` image without CI passing.

### 1. Test coverage & conformance harness

Note: the harness is **Rust**, not TypeScript (`tests/conformance/Cargo.toml`, `tests/conformance/README.md:28`), ~28,900 lines across 12 modules with ~95 test functions.

- **[significant/high]** `.github/workflows/ci.yml:69-126` + `Cargo.toml:3-13` — ~90% of the conformance suite never runs in CI. `tests/conformance` is not a workspace member, so the per-package test shards skip it entirely. The only conformance tests CI runs are the four focused filters inside `scripts/release-smoke:151-169` and `scripts/rollback-smoke:151-169` (`reorg_chaos_drill_conformance_job`, `openapi`, `capability_cutover_evidence`, `dynamic_resolver_profile`). The replay determinism lock (`replay_capability_conformance`), both backfill locks, all `resolution_contract_*`, `primary_names_contract_*`, exact-name, collections, and history contracts (~85 tests in `tests/conformance/src/conformance/`) are gated nowhere. — Fix: add a CI job running `cargo test --manifest-path tests/conformance/Cargo.toml --locked` against the existing Postgres service.
- **Test-matrix mapping vs docs/architecture.md:468-480** (rows with no test anywhere):
  - **[significant/high]** ENSv1 "fuse changes that alter control" — no test at any layer asserts that a burned fuse changes published control/`effective_powers` (and the behavior itself appears unimplemented). Fuse decode/emission is unit-tested only (`crates/adapters/src/ens_v1_unwrapped_authority/tests.rs:891,3054`).
  - **[moderate/high]** ENSv2 "delegate retained after transfer", "shared subregistry creating multiple surfaces", "alias-derived surface with no direct registry entry" — zero hits for delegate/AliasChanged/shared-subregistry in any test (`crates/adapters/src/ens_v2_resolver/` has only 2 tests total, in `events.rs`; no `tests.rs` module).   - **[moderate/high]** DNS/wildcard/offchain row: "imported DNS name", "gasless DNS", "CCIP failure", and "offchain gateway mismatch" have no tests (zero grep hits for gasless; CCIP exists only as a persisted-trace fixture, `tests/conformance/src/conformance/helpers.rs:5324-5510`; offchain appears only as the deferred 404 class, `resolution_and_permissions.rs:2270-2290`).   - **[moderate/medium]** ENSv1 "wrapped owner ≠ registrant" and wrapper expiry/grace edge: adapter-level only (`tests.rs:1769,1938,2029`; `ENS_GRACE_PERIOD_SECS` at `constants.rs:71`); no route/conformance coverage of a wrapped name at all.

  - Operational rows: reorg across authority events, replay determinism (raw + normalized) covered well (`chaos.rs`, `replay.rs`); reorg across execution cache covered at unit level; **[minor/medium]** "proxy implementation change" and "manifest version change" have no dedicated conformance case.

Thin spots: `crates/manifests` (14,362 lines / 48 tests, lowest ratio); large modules with no colocated tests: `crates/adapters/src/ens_v1_subregistry_discovery/checkpoint.rs` (928 lines), `ens_v1_unwrapped_authority/loading/active_emitters.rs` (892), `crates/manifests/src/lib/views/resolver_profiles.rs` (798). ### 5. crates/domain and crates/test-support

- `crates/domain` (258 lines): a single `normalization` module. The pinned normalizer version matches docs exactly (`normalization.rs:3` = `docs/storage.md:78`), and the DNS encode/decode + label rules implement the documented ENSIP-15 single-boundary rule. **[minor/high]** The architecture's "Internal domain model" objects (`docs/architecture.md:225-233`) do not live here — `NameSurface`/`SurfaceBinding`/`TokenLineage`/`Resource` are in `crates/storage/src/identity/types.rs:15-127` (where they match the doc's identity model exactly, including all six `binding_kind` strings), and `ControlVector` etc. live in worker/API layers. `lib.rs:6-8` (`bootstrap_phase()` returning `"phase-1-bootstrap"`) is vestigial. — Fix: either move shared identity enums into domain (as `workstreams.md:21` implies) or correct the ownership map.
- `crates/test-support` (218 lines): solid per-test database harness. **[minor/low]** `TestDatabase::create` leaks the created database if the post-create pool connection fails (`lib.rs:128-138`, no drop guard) — test-infra-only annoyance.

### The 3 biggest gaps in the safety net

1. **The conformance suite is evidence, not a gate.** ~85 of ~95 consumer-contract tests run only when a developer remembers to run them locally with Postgres; CI runs four filters.
2. **Documented fail-closed semantics that are actually fail-open.** Wrapper fuse masking of `effective_powers` is specified in two docs as load-bearing permission truth, has no implementation, and no test would catch it. (Partitioning and object-storage durability are smaller instances of the same failure mode.)
3. **Release supply line is ungated.** The Docker `latest` image publishes on any push to `main` regardless of CI outcome, production compose pulls `latest` by default, and the running services have no healthchecks.
